h1. GNuMIFlux code and config

h2. Overview

This class presents itself as a concrete instance of the @genie::flux::GFluxI@ interface.  The purpose of this case is to read and interpret _flux_ (really, hadron/muon decay) files generated by the various NuMI beam simulations.  These files come in three variants:  @gnumi@ (the initial GEANT3 based simulation), @g4numi@ (a Geant4-based simulation) and @flugg@ (a combination of a modern fluka and geant4).  At the core all three ntuples have a common set of variables, but they differ in the ntuple structure, variable types (float vs. doubles) and element naming (mostly a matter of upper/lower cases).  They also differ to a degree on extra information in the ntuple.  @GNuMIFlux@ attempts to unify the access to all three types by GENIE.  The class has also been used to interpret some re-formulated FNAL Booster Neutrino files (thus negating the overly-specific name given to the class).

The most comprehensive document on the basics of these beam simulations ntuples can be found in "MINOS-DocDB-6316: The NuMI Beam Simulation with Flugg":https://minos-docdb.fnal.gov:440/cgi-bin/ShowDocument?docid=6316.  It is important to note that while entries in the ntuple represent decays of hadrons or muons, they don't reflect _every_ decay.  Generally there is an importance weight attached to each entry that must be accounted for.  Also while each ntuple entry has information about the neutrino from a randomly oriented decay and possibly decays that have the neutrino ray go through two fixed points in space, these are *not* particularly useful for any individual experiment.  The random decays generally miss any detector and the two fixed points (one "near", one "far") fail to illuminate the entirety of the detector (if they are even relevant for that detector, i.e. NOvA vs. MINOS FarDet).

The @GNuMIFlux@ code uses information found in the ntuple to generate neutrino rays over a _flux window_ by evaluating new weights and energies for a ntuple entry based on the position of the random point on the _flux window_ in the beam coordinate system.  This approach was first described in  "MINOS-DocDB-109":https://minos-docdb.fnal.gov:440/cgi-bin/ShowDocument?docid=109.  The _flux window_ should normally be chosen to shadow the entirety of the detector accounting for the orientation of the beam with respect to the detector and any divergence in the beam.

h2. Using the code

The basics of using the code is to create and configure an object:
bc.   genie::flux::GNuMIFlux* gnumi = new GNuMIFlux();
  gnumi->LoadBeamSimData(fluxfname,cfg);
  GFluxI* fdriver = dynamic_cast<GFluxI*>(gnumi);

and then make repeated calls to generate neutrino rays:
bc.   fdriver->GenerateNext();

The two strings passed to @LoadBeamSimData()@ are a pattern that specifies _flux_ files to use and a parameter set name in a XML file.

h3. specifying _flux_ file names

The input files are expected to be ROOT ntuple files of one of the three formats.  Wildcards (normal shell globs, not regexp style) can be used in the filename but not the path part of the string.  All the files must be of the same type (i.e. no mixing of @g4numi@ and @flugg@ files).

h3. XML parameter sets

Experiment specific configurations are encapsulated in &lt;param_set&gt; sections of an XML file; multiple &lt;param_set name="myname"&gt;  instances can exist in a single file as long as each has a unique "myname".  By default @GNuMIFlux@ will look for the file @GNuMIFlux.xml@, but this can be overridden by use of the environment variable @GNUMIFLUXXML@.   The default @GNuMIFlux.xml@ file is found in @$GENIE/src/FluxDrivers/GNuMINtuple@, but alternative locations will be searched if one defines the environment variable @GXMLPATH@; include "." if you want the current location searched before the default.

h2. Configuring a XML param_set

An example parameter set might look like:

bc.   <param_set name="MINOS-FarDet">
    <units> m </units>
    <beamdir type="series">
      <rotation axis="x" units="rad"> +0.057184957 </rotation>
    </beamdir>
    <beampos> (0, 0, 0 ) = ( 0, 0, 735340.0 ) </beampos>
    <window>
      <point coord="det"> -7.000, -9.474, -13.4796 </point>
      <point coord="det">  7.000, -9.474, -13.4796 </point>
      <point coord="det"> -7.000,  7.453, -14.4487 </point>
    </window>
    <enumax> 120. 1.05 1.05 2500000 </enumax>
    <reuse> 10 </reuse>
    <upstreamz> -400.0 </upstreamz>
  </param_set>

The string "MINOS-FarDet" identifies the configuration; this is the string that would get passed as the second argument to @LoadBeamSimData()@.

The top level entities in a &lt;param_set&gt; are described below.  If an entity is specified more than once in a parameter set, the last version trumps any prior settings for that parameter set.  One will find in the default file the  &lt;param_set name="MINOS-NearDet"&gt; specifies the &lt;beamdir&gt; and &lt;beampos&gt; tags multiple times:  *this is for pedagogical purposes*.  All instances give (to within precision) the same result but are there to demonstrate the various alternatives.  Other parameter sets do *not* need to duplicate this; pick one specification and stick with it.

The &lt;beamdir&gt; and &lt;beampos&gt; tags are used together to define the coordinate transformation between the user (i.e. detector) system and the beam system.  The &lt;beamdir&gt; tag must come *before* the &lt;beampos&gt; tag _(n.b. to self: code should rigorously enforce this)_.   These two tags and the &lt;window&gt; are the most critical for an individual detector setup.

h3. &lt;verbose&gt;

This controls how noisy GNuMIFlux is when interpreting the &lt;param_set&gt; and is mostly for debugging purposes.

h3. &lt;units&gt;

The &lt;units&gt; tag identifies the user coordinate system length unit (e.g. "cm", "m" or "meter", etc).  This default is @meter@ (as is the GENIE standard units); the beam _flux_ files are assumed to be in @cm@.  This should match the units used by the detector geometry.  This feature has not been extensively tested and it is probably best to stick with @meter@ if at all possible.

h3. &lt;beamdir&gt;

The &lt;beamdir&gt; defines the rotation matrix between the beam and user coordinates.  There are three ways of specifying this; use the one that is most convenient for the particular setup in question.  If in doubt about the result use the @PrintConfig()@ method to output the final result and verify that it is as you expect.

The first approach is to specify the transformation as a series of rotations around various axes.  Getting this right is easiest if the transformation is a single rotation about one of the primary axes; going beyond this often leads to confusion about order and the correct sign.  An example might look like:
bc.     &lt;beamdir type="series"&gt;
       <rotation axis="x" units="rad"> -0.0582977560 </rotation>
       <!-- additional rotations might follow -->
    </beamdir>

Rotation units for this tag can be either _deg_ or _rad_ and the rotation specifies an axis about which to rotate.  A note about the rotation sign:  the MINOS NearDet sees a downward going beam (mixing _y_ and _z_) which is specified by a negative value as shown here to give the beam a negative _y_ component in the user frame.

The second approach is to specify the beam coordinate's axes in the user system as a set of (&theta;,&phi;) pairs, ala a @GEANT3@ approach:
bc.     <beamdir type="thetaphi3" units="deg">
      ( 90,        0 )  ( 86.65978462612, 90 ) ( 3.34021537388, 270 )
    </beamdir>

Punctuation (i.e. parentheses and commas) aren't relevant and are only given for clarity; tags of this type need six values.

The third and most unambiguous specification is to give the complete 3x3 matrix that would transform a direction vector in beam coordinates into one in user coordinates, ala a @Geant4@ approach:
bc.     <beamdir type="newxyz">
     [ 1  0               0              ]
     [ 0  0.998301167046 -0.058264739543 ]
     [ 0  0.058264739543  0.998301167046 ]
    </beamdir>

A column vector of directions in beam coordinates, when multiplied by this rotation matrix from the left should give the direction in the user system, i.e. @dir@ ~user~ = @Rot@ ~3x3~ * @dir@ ~beam~ .  Again punctuation and layout is irrelevant and shown for only for clarity; tags of this type need nine values.

h3. &lt;beampos&gt;

The &lt;beampos&gt; specification is the second piece needed to complete the coordinate transformations.  There are two variants.  If specified with three values, these are the position of the beam origin in user coordinates:
bc.     <!-- position of beam origin in user coords -->
    <beampos> 1.4828, 60.629225, -1034.72755 </beampos>

Positions are transformed between coordinate systems by:   @xyz@ ~user~ = ( @scale@ @Rot@ ~3x3~ * @xyz@ ~beam~ ) + @xyz@ ~beampos~ where @scale@ is the ratio of user units (e.g. @m@) to beam units (@cm@).

Alternatively it is sometimes easier to know a single point in space in both systems (6 values):
bc.     <!-- position given as ( user coords ) = ( beam coords )                   -->
    <!-- both should be in user coord units, though                            -->
    <!-- convenient for MINOS-NearDet because it is how the survey is reported -->
    <!-- XML config must have already set the rotation matrix                  -->
    <beampos> ( 1.4828, 0.2385, 0.0 ) = ( 0, 0, 1036.48837 ) </beampos>

As with &lt;beamdir&gt; punctuation is irrelevant but useful for clarity.

h3. &lt;window&gt;

The flux window on which neutrino rays will originate is given by specifying three points in user coordinate space.
bc.     <window>
      <point coord="det"> -4.358, -2.903, -0.1835 </point>
      <point coord="det">  4.796, -2.903, -0.1835 </point>
      <point coord="det"> -4.358, 10.377,  0.5923 </point>
    </window>

The order in which they are specified *is* important; the 2nd and 3rd value are used to define vectors relative to the first, so give the common point first (unless you do actually intend to sweep out a rhombus) ala:

In principle is should not matter whether the window is square to the beam direction or the detector as long as it is large enough that rays passing through it completely shadow the volume of interest (including any beam divergence).  _(n.b. to self: still need to empirically verify that window orientation doesn't matter)_

h3. &lt;enumax&gt;

e.g.
bc.  <enumax> 120. 1.05 1.05 2500000 </enumax>

The rejection method used by the de-weighting (importance and x-y over the window) needs an upper estimate of the neutrino energy.  By default it will attempt to guess a conservative value, but CPU cycles might be saved by helping this process along.

This set of 4 parameters set limits on the maximum enu energy:  @lowlimit@, @fudgefactor-E@, and sets limits on max "position" wgt: @fudgefactor-wgt@, @num-points@.  The final reported energy is @max(lowlimit,fudgefactor-E*enumxscan)@ where is @enumxscan@ is the highest energy seen when scanning for a maximum "position" weight limit by generating @num-points@ random rays.  The max "position" weight used is the maximum found scaled up by @fudgefactor-wgt@.

h3. &lt;reuse&gt;

Most entries are rejected (due to importance and x-y weights).  The evaluation process can be sped up some if ntuple entries are reused by choosing a new position on the flux window a number of times.  This increases the very low probability that the same decay entry might actually be accepted twice in a row.  This is very rare and due to the choice of a new position on the flux window results in a different ray direction and energy, but is also undesirable.   This can be tuned based on statistics given after a run by the @PrintConfig()@ method.  Typically a value of 10 is a quite safe choice.

h3. &lt;upstreamz&gt;

By default neutrino rays originate on the specified flux window and GENIE in generating events only considers material in the geometry downstream of this starting position.  Sometimes one would like to leave a flux window in an understood position and orientation and simply start neutrino rays elsewhere.  If @abs(upstreamz)@ <   10^30^ then each ray will be adjusted along its trajectory to start at that @z@ position.

h3. &lt;using_param_set&gt;

A convenient way to build off other configurations is to use the &lt;using_param_set&gt; element:

bc.   <param_set name="MINOS-FarRock">
    <using_param_set> MINOS-FarDet </using_param_set>
    <window>
      <point coord="det"> -15.0000, -17.4612, -13.0224 </point>
      <point coord="det">  15.0000, -17.4612, -13.0224 </point>
      <point coord="det"> -15.0000,  15.4402, -14.9059 </point>
    </window>
  </param_set>

This set takes as its base the "MINOS-FarDet" configuration and modifies the window size.
